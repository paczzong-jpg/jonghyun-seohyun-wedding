# Neon Data API Recipe

## Use When

Use this when the user wants browser CRUD against Neon tables and the project is already configured for Neon Data API plus an auth provider.

## Official Docs To Check

- Data API get started: https://neon.com/docs/data-api/get-started
- Neon Auth and Data API TypeScript SDK: https://neon.com/docs/reference/javascript-sdk
- Data API access control: https://neon.com/docs/data-api/access-control
- Manage Data API: https://neon.com/docs/data-api/manage

## Required Neon Setup

- Neon Data API is in Beta; check current official docs before adding production-critical behavior.
- Data API is enabled for the exact branch/database pair.
- The browser has `VITE_NEON_DATA_API_URL`.
- Signed-in users have a valid JWT. If using public anonymous reads, the configured anonymous role must have explicit `GRANT` and safe policies.
- The table's schema is exposed by the Data API settings.
- The role has `GRANT` privileges for the exact operation.
- Row-Level Security policies allow only intended rows.
- CORS allowed origins include the generated app origin.
- Refresh schema cache after table, column, exposed schema, or policy changes.
- Set a maximum rows limit in Neon settings for broad list endpoints and paginate from the UI.

## Implementation

1. Request env values with the payload in `recipes/neon/README.md`.
2. Copy `recipes/neon/neonClient.ts` to `src/lib/neonClient.ts`.
3. Copy `NeonDataApiTable.tsx` into the feature area and rename the table/columns.
4. Keep queries narrow. Always select explicit columns and add `.limit()`.
5. Treat permission errors as Data API/JWT/`GRANT`/Row-Level Security issues first.

## Example Table Contract

The example expects a `todos` table with these columns:

```sql
id bigint generated by default as identity primary key,
task text not null,
is_complete boolean not null default false,
created_at timestamptz not null default now()
```

For user-private rows, add a user id column and RLS policy based on Neon Data API JWT claims. Do not remove RLS to make the UI work.

## Verification

- Browser code imports `client` from `src/lib/neonClient.ts`.
- Changed browser files do not contain `NEON_API_KEY`, database credentials, or connection strings.
- `client.auth.getSession()` returns a session for private rows.
- `select`, `insert`, `update`, and `delete` each succeed only for rows allowed by Row-Level Security.
- Neon Console Data API settings show the expected branch/database pair, auth provider, exposed schema, CORS allowed origins, and maximum rows.

## Common Wrong Paths

- Using a database connection string in Vite code.
- Using the Node.js serverless driver in browser code.
- Adding a backend route only to bypass missing RLS.
- Forgetting schema cache refresh after table or policy changes.
- Debugging with sandbox shell networking instead of an app route.
